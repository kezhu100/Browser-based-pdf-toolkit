# Architecture

This document describes the current repository architecture and the constraints that future changes must follow.

## Core Rules

1. Keep the existing ToolPlugin architecture.
2. Do not rewrite the app into a different structure.
3. Keep tools modular under `src/tools/plugins`.
4. Reuse shared services under `src/pdf-engine`, `src/pipeline`, and `src/preview-engine`.
5. Keep all processing in the browser.
6. Do not add backend services or server upload flows.
7. Keep the app compatible with GitHub Pages and `HashRouter`.
8. Extend the system in small phases.

## Architectural Layers

### App layer

Responsible for routing, shell layout, and runtime service creation.

Important files:

- `src/app/router.tsx`
- `src/app/AppShell.tsx`
- `src/app/runtimeServices.ts`
- `src/app/pages/ToolsPage.tsx`

### Workspace UI layer

Responsible for tool-family-specific interfaces.

Current workspace families:

- `content`
- `image`
- `pdf`

Workspace routing is handled by:

- `src/components/workspaces/ToolWorkspaceRouter.tsx`
- `src/components/workspaces/config.ts`

### Tool plugin layer

Each user-facing tool is a `ToolPlugin`.

Current implemented tool families:

- content conversion tools
- image conversion tools
- PDF manipulation tools

Current implemented manipulation tools:

- `merge-pdf`
- `split-pdf`
- `reorder-pdf`
- `watermark-pdf`
- `page-numbers-pdf`
- `rotate-pdf`
- `crop-pdf`

Placeholder manipulation tools still exist in the registry, but they are not implemented yet.

### Pipeline layer

The `DocumentPipeline` is used for content-like inputs that can be normalized into a `StandardDocumentModel`.

It is used by:

- Markdown -> PDF
- TXT -> PDF
- HTML -> PDF

It is not the main execution path for PDF binary manipulation.

### Preview layer

The preview renderer creates browser preview HTML from a standard document model.

This supports:

- content workspace preview
- image workspace preview

For the current PDF manipulation workflows, preview is a lightweight summary generated by the plugin rather than a page-thumbnail system.

### PDF engine layer

The PDF engine is the shared boundary for PDF creation and manipulation.

Current responsibilities:

- `generateFromModel()` for content/image export
- `merge()` for PDF manipulation
- `split()` for single-file page splitting
- `reorder()` for page reorder/delete export
- `watermark()` for watermark export
- `pageNumbers()` for page numbering export
- `rotate()` for page rotation
- `crop()` for fixed inset crop export

Current implementation details:

- content/image PDF generation uses the HTML-based generator
- merge uses `pdf-lib` through `src/pdf-engine/adapters/pdfEditAdapter.ts`
- split, reorder, watermarking, page numbering, rotate, and crop also use `pdf-lib` through `src/pdf-engine/adapters/pdfEditAdapter.ts`

Not yet implemented in the engine:

- none currently

## ToolPlugin Contract

`ToolPlugin` is the central extension point.

Each tool declares:

- `id`
- `name`
- `description`
- `category`
- `inputTypes`
- `capabilities`
- `settingsSchemaName`
- `run(context)`
- optional `previewAdapter`

Registry location:

- `src/tools/registry.ts`

## Execution Paths

### Content and image conversion path

```text
SourceDocument
  -> ToolPlugin.run()
  -> DocumentPipeline
  -> StandardDocumentModel
  -> PreviewRenderer
  -> UnifiedPdfEngine.generateFromModel()
  -> Browser download
```

### PDF manipulation path

```text
PDF SourceDocument[]
  -> merge/split/reorder/watermark/page-numbers/rotate/crop ToolPlugin.run()
  -> UnifiedPdfEngine manipulation method
  -> pdfEditAdapter
  -> pdf-lib edit in browser
  -> Browser download
```

This distinction is important:

- content/image tools should continue to reuse the document pipeline
- PDF manipulation tools should continue to extend the `pdf-engine` path

## Workspace Design

### Content workspace

Supports:

- tool selection for markdown/txt/html
- local file upload
- editor input
- automatic preview
- explicit PDF export
- export settings

### Image workspace

Supports:

- local image file upload
- multi-image handling
- automatic preview
- explicit PDF export
- export settings reuse

### PDF workspace

Current Phase 10.1 scope:

- tool selection for `merge-pdf`
- multi-file PDF input
- validation
- lightweight merge summary preview
- explicit merge/export action

Current Phase 10.2 additions:

- split single PDF into one file per page
- rotate all pages of one PDF by 90 / 180 / 270 degrees
- keep preview as a lightweight summary rather than a page thumbnail editor

Not included yet:

- page thumbnails
- page-level manipulation UI

## Routing and Deployment Constraints

The app must remain statically deployable.

Requirements:

- `HashRouter`
- GitHub Pages compatibility
- no backend fallback

The current tools route is still served under:

- `#/tools`

## Dependency Constraints

Dependencies should remain minimal and browser-compatible.

Current PDF-related libraries:

- `html2pdf.js` for HTML-based export flow
- `pdf-lib` for browser-side PDF manipulation

When adding future PDF manipulation features, integrate them through the `pdf-engine` layer rather than UI components.

## Development Guidance

When extending the project:

1. inspect the existing plugin, workspace, and engine layers first
2. extend the nearest existing pattern instead of inventing a parallel architecture
3. keep changes small and phase-oriented
4. document implemented status accurately
5. do not claim placeholder tools are complete
